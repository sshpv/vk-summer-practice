# Практический проект VK Education: сравнение визуально-языковых моделей (VLM) на бенчмарках GQA-ru и MMBench-ru от deepvk

## Название проекта
Сравнительный анализ производительности визуально-языковых моделей на бенчмарках GQA-ru и MMBench-ru

## Цель проекта
Сравнить производительность существующих визуально-языковых моделей — baseline-решения deepvk (`llava-saiga-8b`) и современной компактной мультиязычной VLM (`Qwen2.5-VL-3B-Instruct`) — на открытых русскоязычных бенчмарках GQA-ru и MMBench-ru, и определить, какая модель показывает более высокое качество на каждом из бенчмарков.

### Почему выбран именно такой вариант проекта
Формулировка задания прямо допускает два пути: обучить собственную модель с нуля **или** «провести проверку производительности различных моделей» на предложенных бенчмарках. Для этого проекта был осознанно выбран **второй вариант** — сравнение существующих моделей без дообучения — по следующим причинам:

- **Ограничение по времени.** Полноценное дообучение (даже LoRA) VLM с последующей валидацией на двух бенчмарках требует существенно больше вычислительного времени, чем позволяли сроки практики.
- **Ограничение по вычислительным ресурсам.** Доступное железо — бесплатные GPU в Google Colab (T4, 16 GB) и Kaggle (T4 x2) — не гарантирует стабильного и достаточно быстрого времени на полный цикл «дообучение + повторная оценка» без риска не уложиться в дедлайн.
- **Смысловая полнота даже без дообучения.** Задание явно требует «проверку производительности различных моделей», а не обязательно создание новой — сравнение готовых моделей само по себе является полноценным, воспроизводимым и инфмормативным исследованием.

Таким образом, шаг с LoRA-дообучением (изначально заложенный в задачах как опциональный, «при наличии ресурсов») в финальную версию проекта не вошёл — вместо него сделан более тщательный, воспроизводимый и корректно задокументированный процесс сравнения двух готовых моделей.

## Задачи (финальная версия)
1. Изучить открытые датасеты и baseline-модели из коллекции deepvk Vision-Language Modeling (`GQA-ru`, `MMBench-ru`, `llava-saiga-8b`).
2. Развернуть пайплайн оценки моделей на базе библиотеки `lmms-eval`.
3. Получить baseline-метрики модели deepvk (`llava-saiga-8b`) на GQA-ru и MMBench-ru.
4. Оценить zero-shot производительность современной компактной мультиязычной VLM (`Qwen2.5-VL-3B-Instruct`) на тех же бенчмарках и сравнить с baseline.
5. Задокументировать методологию, использованные данные, технические решения и результаты сравнения моделей.

## Ожидаемые результаты (финальная версия)
- Таблица сравнения метрик (ExactMatch / GPTEvalScore) для обеих моделей на GQA-ru и MMBench-ru.
- Итоговый вывод — какая модель показывает более высокое качество на каждой задаче и почему.
- Воспроизводимый Jupyter/Colab/Kaggle-ноутбук с пайплайном оценки.
- Файл с подробным описанием решения (этот документ): постановка задачи, использованные данные, ход эксперимента, технические сложности и их решения, метрики, выводы.

## Способы и средства достижения результата

**Среда вычислений:** Google Colab (T4 GPU, 16 GB VRAM) и Kaggle Notebooks (T4 x2, 30 ч GPU-квоты в неделю) — использовались оба варианта из-за нестабильности бесплатных сессий Colab при длительных прогонах; финальные полные прогоны выполнены на Kaggle, где интерактивная сессия устойчивее держится в фоне.

**Стек инструментов:**

| Задача | Инструмент |

| Оценка моделей на бенчмарках | [`lmms-eval`](https://github.com/EvolvingLMMs-Lab/lmms-eval) |
| Загрузка датасетов/моделей | `datasets`, `transformers`, `huggingface_hub` (Hugging Face) |

| Среда выполнения | Google Colab / Kaggle Notebooks (Python, GPU T4) |

**Модели для сравнения:**
1. `deepvk/llava-saiga-8b` (8B) — baseline от deepvk
2. `Qwen/Qwen2.5-VL-3B-Instruct` — современная компактная мультиязычная VLM для сравнения

## Использование открытых данных VK

Оба бенчмарка взяты напрямую из открытой коллекции deepvk на Hugging Face (`huggingface.co/collections/deepvk/vision-language-modeling`) и подключены как встроенные задачи фреймворка `lmms-eval` (`gqa-ru` и `mmbench_ru_dev`) — датасеты скачиваются автоматически из репозиториев deepvk при первом запуске задачи, без ручного скачивания или предобработки.

### GQA-ru
Переведённая на русский версия бенчмарка GQA (визуальные вопросы-ответы).

- **Test split (testdev):** 398 изображений, 12 216 вопросов
- **Ключевые поля:** `question`, `answer`, `fullAnswer`, `imageId`, `isBalanced`, `groups`, `types` (structural / semantic / detailed), `annotations`
- **Метрика:** ExactMatch — сгенерированный моделью ответ должен точно совпасть с эталонным

### MMBench-ru
Переведённая версия MMBench, multiple-choice вопросы по изображениям.

- **Размер:** 3 910 примеров
- **Формат:** вопрос + варианты ответов (2–4 опции), модель выбирает букву
- **Метрика:** GPTEvalScore — без OpenAI API ключа сводится к точному совпадению выбранной буквы с эталоном (использованный в проекте вариант)

### Ограничение по объёму выборки
Из-за жёстких сроков практики и ограниченного времени бесплатных GPU-сессий, оценка проводилась не на полном test-сплите (12 216 + 3 910 примеров суммарно), а на **случайной подвыборке по 300 примеров на каждый бенчмарк** (`--limit 300`), **одинаковой для обеих моделей** (один и тот же seed → одна и та же случайная подвыборка для честного сравнения). Этого объёма достаточно для содержательного сопоставления моделей в рамках учебного проекта — соответствующий доверительный интервал (Stderr) приведён вместе с метриками ниже.

## Ход реализации и технические решения

Пайплайн оценки строился на официальном фреймворке `lmms-eval`, который используют для оценки VLM. В процессе настройки окружения возник ряд несовместимостей между зафиксированными в задании компонентами (Colab/Kaggle, свежая версия `lmms-eval`, конкретные модели) — они были продиагностированы и решены:

- **Конфликт версий `transformers` между моделями.** `Qwen2.5-VL` требует `transformers>=4.49.0` (класс `Qwen2_5_VLForConditionalGeneration` появился только в этой версии), тогда как чекпоинт `deepvk/llava-saiga-8b` совместим только со старым форматом препроцессора (без `patch_size`/`vision_feature_select_strategy`, обязательных в свежих версиях `transformers`, начиная примерно с 4.47). Решение: два разных запуска в рамках одной сессии — `transformers==4.43.4` для deepvk, `transformers>=4.49.0` для Qwen2.5-VL, с переустановкой пакета и перезапуском runtime между ними.
- **Обёртка модели без chat-шаблона.** У `llava-saiga-8b` отсутствует чат-шаблон в конфиге токенизатора — потребовался флаг `--force_simple` в `lmms-eval`, переключающий модель на «простой» (не-чат) режим генерации.
- **Отсутствующие поля препроцессора.** Пришлось точечно пропатчить обёртку `llava_hf` в `lmms-eval`, чтобы она проставляла `patch_size` и `vision_feature_select_strategy` из конфига самой модели, если они отсутствуют в конфиге препроцессора.
- **Несовместимые именования параметров.** Обёртка `qwen2_5_vl` не принимает параметр `dtype` в `model_args` (в отличие от `llava_hf`, где он поддерживается) — потребовалась разная конфигурация запуска для каждой модели.
- **Нестабильность бесплатных GPU-сессий.** Из-за многочасовых прогонов и разрывов сессий Colab финальные полные прогоны выполнены на Kaggle Notebooks с `Save Version → Save & Run All` (фоновый запуск, не зависящий от активности браузера).

Итоговые команды запуска (после настройки окружения):

```bash
# deepvk/llava-saiga-8b — GQA-ru и MMBench-ru
python -m lmms_eval \
  --model llava_hf \
  --model_args pretrained=deepvk/llava-saiga-8b,device_map=auto,dtype=bfloat16 \
  --tasks gqa-ru,mmbench_ru_dev \
  --batch_size 1 \
  --force_simple \
  --limit 300 \
  --log_samples --log_samples_suffix deepvk \
  --output_path ./logs/

# Qwen2.5-VL-3B-Instruct — GQA-ru и MMBench-ru
python -m lmms_eval \
  --model qwen2_5_vl \
  --model_args pretrained=Qwen/Qwen2.5-VL-3B-Instruct,attn_implementation=sdpa,device_map=auto \
  --tasks gqa-ru,mmbench_ru_dev \
  --batch_size 1 \
  --limit 300 \
  --log_samples --log_samples_suffix qwen25vl \
  --output_path ./logs/
```

## Результаты

| Модель | GQA-ru, ExactMatch (n=300) | MMBench-ru, GPTEvalScore (n=300) |
|---|---|---|
| `Qwen/Qwen2.5-VL-3B-Instruct` | 39.7% (±2.8 п.п.) | **78.3** |
| `deepvk/llava-saiga-8b` | **45.0%** (±2.9 п.п.) | 69.0 |

*Полные логи (results.json, сэмплы предсказаний по каждому вопросу) приложены в архиве `results_gqa_mmbench_ru.zip`.*

### Выводы

Однозначного лидера по обеим задачам не выявлено — у каждой модели своя сильная сторона:

- **На GQA-ru (открытые вопросы-ответы по изображению) сильнее `deepvk/llava-saiga-8b`** — 45.0% против 39.7% у Qwen2.5-VL. Вероятное объяснение: `llava-saiga-8b` дообучалась именно на русскоязычных инструкциях (`LLaVA-Instruct-ru`) с учётом специфики формата ответа «одним словом», принятого в GQA, тогда как Qwen2.5-VL — более универсальная мультиязычная модель без узкой русскоязычной специализации под этот формат.
- **На MMBench-ru (выбор варианта A/B/C/D) сильнее `Qwen2.5-VL-3B-Instruct`** — 78.3 против 69.0. Вероятное объяснение: формат multiple-choice лучше поддаётся более крупной и свежей базе предобучения Qwen2.5-VL, а также инструктивная тонкая настройка этой модели явно ориентирована на строгое следование формату ответа.

Таким образом, выбор «лучшей» модели зависит от прикладной задачи: для свободной генерации ответов на русском языке предпочтительнее `llava-saiga-8b`, для задач с фиксированным набором вариантов ответа — `Qwen2.5-VL-3B-Instruct`.

### Ограничения исследования
- Оценка проведена на подвыборке 300 примеров на бенчмарк (а не на полном test-сплите) из-за ограничений по времени и вычислительным ресурсам — результаты стоит воспринимать как индикативные, а не как исчерпывающую оценку.
- Метрика MMBench-ru использована в упрощённом варианте (точное совпадение буквы) без OpenAI API — это стандартный fallback-режим `lmms-eval` при отсутствии API-ключа, соответствующий условиям задания (использование только открытых средств).
- Дообучение моделей в рамках этой версии проекта не проводилось (см. раздел «Почему выбран именно такой вариант проекта» выше).
